# 서사형 세계모델 엔진 재설계안

작성일: 2026-03-10  
목적: 최신 기술 동향을 반영해, 지정학/거시 이벤트를 구조 가설과 투자 행동으로 변환하는 엔진의 현실적 설계 기준을 재정의한다.

---

## 0. 온보딩 기준

이 문서는 더 이상 "빈 상태의 신규 제품"을 가정하지 않는다.

2026-03-10 시점의 JARVIS는 이미 다음을 갖춘 상태다.

- 세션 기반 요청 진입
- capability/stage 기반 오케스트레이션
- profile-aware research -> dossier/brief 생성
- monitor/watcher -> follow-up -> action proposal 흐름
- memory note 기반 개인화
- execution option + approval gate
- Assistant / Dossier / Watchers / Action Center UI 표면

즉, 서사형 세계모델 엔진은 별도 신규 제품이 아니라  
`현재 JARVIS 세션 운영체계 내부에 삽입되는 추론 코어`로 설계해야 한다.

### 0.1 현재 코드베이스에서 확인된 실제 중심선

실제 시스템의 중심은 문서보다 코드에 더 잘 드러난다.

- 백엔드 엔트리: Fastify 서버 + 라우트 등록 + 워커/알림 초기화
- 세션 엔진: `jarvis/request-service.ts`, `jarvis/stages.ts`
- 리서치 산출물: `jarvis/research.ts`, `routes/dossiers.ts`
- 메모리/개인화: `jarvis/memory-context.ts`, `memory_notes`
- 모니터링: `jarvis/watcher-worker.ts`, `routes/watchers.ts`
- 액션/승인: `action_proposals`, `Action Center`, execution option 흐름
- 실제 저장 스키마: `docs/db-schema-v1.sql`보다 `store/postgres/initializer.ts`가 최신 상태에 가깝다

즉 설계 기준은 오래된 정적 문서가 아니라  
`현재 작동 중인 session -> dossier -> watcher -> action proposal` 흐름이어야 한다.

### 0.2 이 설계가 접합되어야 하는 현재 표면

세계모델 엔진은 다음 표면을 대체하지 않는다.

- Assistant를 없애지 않는다
- Dossier를 없애지 않는다
- Watcher를 없애지 않는다
- Action Center를 없애지 않는다

대신 다음을 강화한다.

- Dossier의 근거/충돌 요약을 `가설/반대가설/병목/무효화 조건`까지 확장
- Watcher의 change detection을 `구조 변화 감지`로 고도화
- Action Proposal을 단순 이벤트 후속조치가 아니라 `세계모델 상태 변화 기반 제안`으로 고도화
- Memory를 사용자 선호 저장소로 유지하되, 세계모델과 엄격히 분리

### 0.3 초기 가정 수정

처음 문서에서 암묵적으로 가정했던 것과 달리, 현재 레포는 이미 상당한 운영 표면을 갖고 있다.

따라서 올바른 질문은:

- "이 세계모델 엔진을 새로 만들까?"가 아니라
- "현재 세션/도시에/워처/액션 흐름의 어느 지점에 삽입할까?"다

정답은 다음에 가깝다.

- 입력: research request / watcher run / monitor hit
- 내부 코어: extraction -> semantic normalization -> state update -> hypothesis ledger
- 출력: dossier enrichment / briefing enrichment / follow-up classification / action proposal justification

## 1. 전제

이 시스템의 목적은 "세상의 모든 데이터를 실시간으로 이해하는 범용 AI"가 아니다.

정확한 목표는 다음과 같다.

- 제한된 도메인에서
- 다중 소스 데이터를 실시간 또는 준실시간으로 수집하고
- 그것을 엔티티/이벤트/관측/제약으로 정규화한 뒤
- 경쟁 가설을 유지하면서
- 반증 조건과 추적 지표까지 제시하는
- 상태 추정형 분석 엔진을 만든다

즉, 이 시스템의 본체는 검색기가 아니라 `가설 원장을 가진 세계모델`이다.

## 2. 2026-03-10 기준 핵심 기술 판단

### 2.1 GraphRAG는 중요한 기술이지만, 코어 엔진은 아니다

최근 GraphRAG 계열은 분명히 성숙했다.

- Microsoft GraphRAG
- DRIFT Search
- LazyGraphRAG
- Neo4j GraphRAG

이 계열은 "문서 집합 전체에서 전역 구조를 요약하고, 지역/전역 검색을 혼합해 답변 품질과 비용을 개선"하는 데 강하다.

하지만 GraphRAG는 기본적으로 `검색/컨텍스트 구성 기술`이다.

따라서 다음은 잘하지만:

- 전역 질문 대응
- 문서 집합 구조화
- 관련 커뮤니티/서브그래프 선택
- 근거 검색과 설명 가능성 강화

다음은 직접 해결하지 못한다:

- 병목 추정
- 숨은 유인 추정
- 경쟁 가설 유지
- 가설 무효화
- 사후 적중률 보정

결론:

- GraphRAG는 `세계모델의 검색 인터페이스`로 채택
- 그러나 `세계모델 자체`로 오해하면 안 된다

### 2.2 최신 메모리 계열의 핵심은 "더 많이 기억"이 아니라 "어떻게 갱신하고 잊는가"다

2025~2026의 흐름은 분명하다.

- Graphiti/Zep: 시간 인지형 그래프 메모리
- Mem0/OpenMemory: 장기 메모리 계층
- LangGraph v1: durable execution + 장기 메모리
- Memory in the Age of AI Agents: 메모리를 1급 primitive로 재정의
- AI Agents Need Memory Control Over More Context: transcript replay보다 bounded state가 중요
- MemoryArena: 다중 세션 에이전트 메모리 평가

핵심 변화:

- 긴 컨텍스트를 그대로 넣는 방식은 한계가 뚜렷하다
- 메모리는 저장보다 `선택`, `갱신`, `무효화`, `압축`, `회수`가 중요하다
- 에이전트 메모리는 문서 저장소보다 `변화하는 사실과 관계의 상태 저장소`에 가까워지고 있다

결론:

- 우리 설계는 문서 중심이 아니라 `양방향 시간성(valid time + recorded time)`을 가진 사실/관계 저장소를 가져야 한다
- 세션 메모리와 세계모델 메모리는 분리해야 한다

### 2.3 Structured Output은 추출 안정성을 크게 높였지만, 진실성은 보장하지 않는다

OpenAI Structured Outputs와 Gemini Structured Output은 2026년 현재 매우 유용하다.

이걸 통해:

- 엔티티 추출
- 이벤트 추출
- 관측값 추출
- 관계 타입 분류
- 가설/반대가설 포맷화

를 매우 안정적으로 JSON Schema에 맞춰 받을 수 있다.

하지만 여전히 남는 문제:

- 값 자체의 사실성
- 시간축 해석 오류
- 관계의 과잉 일반화
- 숫자 단위 혼동

결론:

- extraction 단계는 반드시 schema-first로 간다
- 그러나 extraction 결과는 바로 truth가 아니라 `candidate fact`로 취급한다

### 2.4 최신 표준은 "의미 계층"과 "프로토콜 계층"이 분리되는 방향이다

현재 관련 흐름은 두 갈래다.

- 의미/제약: RDF, OWL, SKOS, SHACL 1.2
- 도구/에이전트 연결: MCP, A2A, Agent Spec 계열

의미:

- SKOS: 용어 체계/분류 체계
- SHACL 1.2: 그래프/의미 데이터 검증
- MCP: 도구/리소스 표준 연결
- A2A: 에이전트 간 통신/위임 표준

결론:

- 의미 계층은 `SKOS + SHACL 우선`, OWL은 나중
- 에이전트 연결 계층은 `MCP 우선`, 다중 시스템 협업이 필요해질 때 `A2A` 검토

### 2.5 실시간 분석 스택은 "전부 LLM"이 아니라 스트리밍 계산이 핵심이다

현실적으로 중요한 것은 다음이다.

- Kafka / Kafka Connect
- Debezium CDC
- Flink 2.x
- Materialize의 증분 계산
- TimescaleDB Hypercore

이 계열이 필요한 이유:

- 뉴스/가격/지표/이벤트는 들어오는 즉시 상태를 갱신해야 한다
- 매번 전체 재계산하면 비용과 지연이 감당되지 않는다
- "현재 살아 있는 가설"에 연결된 부분만 증분 갱신해야 한다

결론:

- 세계모델은 반드시 `event-driven incremental update` 구조여야 한다

### 2.6 평가 스택은 선택이 아니라 필수다

2025~2026의 흐름에서 분명해진 점이 하나 있다.

- 추론 엔진은 "좋은 답변"만으로는 개선되지 않는다
- extraction, linking, hypothesis scoring, invalidation 판단을 각각 따로 평가해야 한다
- trace 기반 평가와 사후 outcome 평가가 모두 필요하다

관련 흐름:

- OpenAI Evals
- LangSmith Evaluation
- Arize Phoenix Eval / Trace
- DSPy 계열의 programmatic optimization

결론:

- V1부터 `offline eval + online trace + outcome backtest`를 같이 설계한다
- 없으면 시스템은 그럴듯한 서사를 반복할 뿐, 어디서 틀렸는지 학습하지 못한다

## 3. 최종 설계 원칙

### 원칙 1. 메모리와 세계모델을 분리한다

- 메모리: 사용자/세션/작업 연속성
- 세계모델: 외부 세계의 엔티티, 사건, 제약, 가설

### 원칙 2. 사실과 추론을 분리한다

- observed
- extracted
- inferred
- speculative
- disputed
- invalidated

각 상태는 별도 필드와 별도 점수를 가져야 한다.

### 원칙 3. 문서보다 화살표를 저장한다

핵심 단위:

- 사건
- 관측
- 제약
- 메커니즘
- 가설
- 무효화 조건

### 원칙 4. 반드시 반대가설을 같이 운영한다

좋은 서사 엔진의 조건은 정답률보다 `잘 틀리는 능력`이다.

즉:

- 주 가설
- 반대가설
- 둘을 구분하는 다음 관측

를 같이 저장해야 한다.

### 원칙 5. "예상됐지만 안 나온 신호"를 핵심 증거로 쓴다

구조적 추론에서 가장 중요한 정보는 종종 부재다.

예:

- 리스크는 커졌는데 운임이 안 오른다
- 공급 불안인데 계약 체결이 안 나온다
- 뉴스는 강한데 실적 가이던스가 안 움직인다

이 부재 정보가 가설 약화의 핵심이다.

## 4. 추천 아키텍처

### 4.1 Ingestion Plane

역할:

- 뉴스/정책/공시/가격/물류/재고/실적 데이터 수집
- 스트리밍 또는 배치 ingest

권장:

- 현재 레포 기준으로는 Postgres 중심 유지
- 추후 입력량이 늘면 Kafka Connect / Debezium / Materialize 도입

### 4.2 Extraction Plane

역할:

- 원문 -> 엔티티/이벤트/관측/클레임/수치/시간/단위 추출

권장:

- Structured Output 기반
- 추출 결과는 schema-first
- 단위/시간/통화 검증은 SHACL 또는 별도 validator

### 4.3 Semantic Plane

역할:

- canonical entity
- alias resolution
- event normalization
- typed relation

권장:

- 가벼운 ontology: SKOS + custom relation taxonomy
- 무거운 OWL 추론은 초기 범위에서 제외

### 4.4 State Plane

역할:

- 숨은 상태 추정
- 병목 추적
- 계약/물류/금융/정책 채널별 상태값 유지

예시 상태 변수:

- route_risk
- freight_pressure
- insurance_pressure
- contract_urgency
- capex_support_probability
- inflation_passthrough_risk
- rate_repricing_pressure

### 4.5 Hypothesis Ledger

역할:

- 주 가설
- 반대가설
- 신뢰도
- 무효화 조건
- 다음 추적 신호

이 계층이 시스템의 본체다.

### 4.6 Retrieval / Dossier Plane

역할:

- GraphRAG / hybrid retrieval
- 전역/지역 설명 생성
- 근거 앵커와 구조 설명 조합

권장:

- GraphRAG는 이 계층에서 사용
- 검색 보강용이지 코어 상태 추론 엔진으로 쓰지 않는다

### 4.7 Evaluation / Outcome Plane

역할:

- 나중에 가설이 맞았는지 평가
- 어떤 링크가 약했는지 기록
- memory / hypothesis calibration

권장:

- trace + evaluator + outcome journal
- 장기적으로는 내부 MemoryArena형 벤치 필요

### 4.8 Narrative Compiler Plane

역할:

- 세계모델과 가설 원장을 사람이 읽는 분석문으로 변환
- 모순 제시 -> 표면 설명 -> 병목 -> 대체 가설 -> 반대가설 -> 추적 지표 -> 행동 항목 순서로 컴파일

중요:

- 이 계층은 추론 엔진이 아니다
- 이미 계산된 상태와 가설을 사람이 이해하는 서사로 정렬하는 계층이다
- 따라서 서사 품질이 높아도, 아래 가설/상태 계층이 빈약하면 전체 시스템은 쉽게 체리피킹으로 무너진다

## 5. 저장 전략

### V1 권장 저장소

- Postgres
- pgvector
- 시계열 저장은 Timescale 계열 검토
- JSONB + 정규 테이블 혼합

이유:

- 현재 레포와 가장 잘 맞는다
- 운영 복잡도가 낮다
- hypothesis ledger와 outcome journal을 빠르게 붙일 수 있다

### 그래프 저장소 도입 시점

다음 조건이 충족될 때 별도 그래프 스토어 검토:

- 다중 hop explanation 요구가 크게 증가
- subgraph retrieval 비용이 높아짐
- multi-agent shared context graph 수요가 증가

후보:

- Neo4j
- Apache AGE
- Graphiti 계열 패턴 차용

초기 판단:

- V1은 별도 그래프 DB 없이 시작하는 편이 낫다
- 먼저 부족한 것은 graph query 성능이 아니라 ontology discipline과 hypothesis ledger다

## 6. 첫 구현 범위

도메인은 반드시 좁힌다.

- 중동 지정학
- LNG
- 해운/보험
- 인플레/금리
- 수혜 산업/종목

질문도 좁힌다.

- "이 이벤트는 진짜 구조 재편인가, 아니면 헤드라인 노이즈인가?"

필수 데이터만 붙인다.

- 지정학 뉴스
- 정책/제재 문서
- LNG 가격
- 원유 가격
- 금리/달러
- 운임/물류 프록시
- 계약 체결 뉴스
- 터미널 증설/장애
- 기업 IR/실적콜

## 7. 최종 판단

가능하다.

하지만 조건이 있다.

- 전 세계 범용 엔진으로 시작하면 실패한다
- 한정된 세계를 추적하는 상태 추정기로 시작해야 한다
- GraphRAG, memory, agent runtime, streaming, ontology는 모두 필요하지만 각자 역할이 다르다
- 코어는 `hypothesis ledger + temporal state model + invalidation engine`이다

한 줄 요약:

이 시스템은 "많이 기억하는 AI"가 아니라,  
`시간에 따라 변하는 세계를 경쟁 가설로 추적하고 반증하는 분석 운영체계`로 설계해야 한다.
