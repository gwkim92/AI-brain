# 자율 레이더 스캐너 구현 계획

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** JARVIS가 외부 신호를 주기적으로 직접 수집하고, 자동으로 `ingest -> evaluate -> promote`를 수행해 진짜 광역 감시형 레이더로 동작하게 만든다.

**Architecture:** 현재 radar는 `수동 ingest + 수동 evaluate` 엔진까지만 구현되어 있다. V1 확장은 `피드 소스 카탈로그`, `수집/정규화 worker`, `최근 item 기반 이벤트 병합`, `자동 평가/승급 worker`, `운영 상태/킬스위치`를 기존 `radar -> watcher/dossier/action -> world-model outcome` 흐름 앞단에 삽입하는 방식으로 간다. 핵심은 “세상을 직접 훑는 worker”와 “여러 신호를 하나의 event로 합치는 계층”을 추가하는 것이다.

**Tech Stack:** Fastify, Postgres, 기존 `startWorkerSupervisor`, RSS/Atom/JSON fetch, 기존 radar/world-model 저장소, Vitest, pnpm

## 구현 우선순위 판단

이 플랜의 핵심은 3단이다.

1. `자동 수집기`를 붙인다.
- 지금 가장 큰 공백이다.
- 이게 없으면 JARVIS는 여전히 외부에서 밀어넣은 신호만 판단한다.

2. `이벤트 병합`을 붙인다.
- 지금은 사실상 item 1개가 event 1개라서 “세상을 훑는다”는 느낌이 약하다.
- 여러 기사/수치/공시를 하나의 사건으로 합쳐야 구조성 판단이 올라간다.

3. `자동 evaluate + 자동 promote`를 worker 안으로 옮긴다.
- 이게 붙어야 비로소 HTTP route 없이도 레이더가 돌아간다.

## 어떤 데이터를 V1에서 자동 수집할 것인가

V1은 “세계 전체”가 아니라 아래 4계층으로 시작한다.

- `tier_0 official`
  - 정부/규제/공식 changelog/SEC/중앙은행 공지
- `tier_1 trusted document feed`
  - 허가된 RSS/Atom/JSON 문서 피드
- `tier_2 secondary analysis`
  - 연구/산업 블로그
- `synthetic/internal`
  - ops-policy, 내부 watcher-derived signals

시장 수치형 데이터는 별도 유료 커넥터를 붙이기 전까지는 `manual_metric` 또는 제한된 JSON feed로만 받는다. 처음부터 “모든 시장 데이터 실시간 ingest”를 목표로 잡으면 구현 범위가 무너진다.

## 완료 조건

다음 네 조건을 만족하면 “JARVIS가 스스로 세상을 훑는다”고 말할 수 있다.

- 운영자 호출 없이 주기적으로 feed를 fetch한다.
- 새 item을 자동으로 event/posterior/recommendation으로 만든다.
- 구조성이 높은 event는 watcher/dossier/action으로 자동 승급된다.
